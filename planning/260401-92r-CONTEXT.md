<decisions>
## Implementation Decisions

### Container-Orchestrierung
- Docker API direkt via Python docker SDK (kein Coolify, kein Portainer)
- Python FastAPI Webhook Receiver ruft docker.containers.run() auf
- Kein extra Orchestrierungs-Tool — maximal simpel

### Credential-Sicherheit
- LLM-Key + Bot-Token encrypted in Supabase Vault gespeichert
- NIE auf VPS-Disk geschrieben
- Webhook-Payload enthält Keys (HTTPS + Webhook-Secret gesichert)
- Python Receiver gibt Keys als Docker ENV-Variablen weiter — nur im Container-Memory
- ZeroClaw liest Keys aus ENV, LLM-Agent sieht nie den Raw-Key

### ZeroClaw Docker Image
- Dockerfile baut aus b0xtch/zeroclaw Source (cargo build --release)
- Multi-stage Build: Builder (Rust) → Runtime (minimal Debian/Alpine)
- ARM64 kompatibel (Hetzner CAX21)
- Einmal bauen, alle Container nutzen dasselbe Image

### Provisioning-Trigger
- Test-Phase: Sofort bei Formular-Submit
- Produktion: Nach Stripe Webhook → user.status = 'paid' in Supabase → dann Edge Function triggert Provisioning

### Claude's Discretion
- Port-Zuweisung: Base 42000 + auto-increment via registry file
- Webhook-Secret: Zufälliger 32-Byte Hex-String, in Supabase ENV + VPS ENV
- Container-Name Schema: fc-{user_id}-{agent_id}
- Workspace-Volume: /opt/focuscall/workspaces/{user_id}/{agent_id}/ (nur Workspace, keine Keys)

</decisions>

<specifics>
## Specific Ideas

**Flow komplett:**
1. User füllt Formular aus (LLM Key + Telegram Bot Token)
2. React Frontend → Supabase Edge Function `provision-agent`
3. Edge Function: Key in Supabase Vault speichern, Webhook an VPS senden
4. VPS FastAPI: Webhook validieren, docker.containers.run() aufrufen, Keys als ENV
5. Container startet, ZeroClaw daemon läuft, Bot antwortet

**Dateien die entstehen sollen:**
- `~/focuscall-architecture/provisioning/edge-function.ts` — Supabase Edge Function
