**edge-function.ts** (Supabase Edge Function, Deno/TypeScript):
- Import: serve from Deno std, createClient from @supabase/supabase-js
- Empfaengt POST mit JSON: { user_id, agent_id, llm_key, bot_token, llm_provider }
- Speichert llm_key und bot_token in Supabase Vault via vault.secrets.create() — pgsodium-basiert
- Erstellt HMAC-SHA256 Signatur ueber den Payload (user_id + agent_id + timestamp) mit WEBHOOK_SECRET aus Deno.env
- Sendet POST an VPS_WEBHOOK_URL/provision mit Headers: Content-Type application/json, X-Timestamp, X-Signature
- Payload an VPS enthaelt: user_id, agent_id, llm_key, bot_token, llm_provider (Keys im Klartext, gesichert durch HTTPS + HMAC)
- Error Handling: try/catch, spezifische HTTP Status Codes (400 missing fields, 500 vault/webhook error)
- Env vars benoetigt: SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, VPS_WEBHOOK_URL, WEBHOOK_SECRET

**webhook-receiver.py** (FastAPI, Python 3.11):
- FastAPI app mit POST /provision Endpoint
- HMAC-SHA256 Signatur-Validierung: berechnet HMAC ueber request body mit WEBHOOK_SECRET, vergleicht mit X-Signature Header
- Timestamp-Check: lehnt Requests aelter als 5 Minuten ab (Replay-Schutz)
- Ruft provision_container() aus provision.py auf (async, im Background via BackgroundTasks)
- Gibt sofort 202 Accepted zurueck mit { status: "provisioning", user_id, agent_id }
- DELETE /provision/{user_id}/{agent_id} Endpoint fuer Deprovision
- Health-Check GET /health
- Logging: structlog oder Standard-Python-Logging
- Konfiguration via Environment: WEBHOOK_SECRET, REGISTRY_PATH=/opt/focuscall/registry.json, WORKSPACE_BASE=/opt/focuscall/workspaces
- uvicorn Entry-Point am Ende: if __name__ == "__main__": uvicorn.run()

**provision.py** (Docker SDK, Python 3.11):
- import docker, json, shutil, string, pathlib, logging
- CONFIG_TEMPLATE_PATH zeigt auf config.toml.tmpl im selben Verzeichnis (fuer Deployment: /opt/focuscall/config.toml.tmpl)
- WORKSPACE_BASE = /opt/focuscall/workspaces
- REGISTRY_PATH = /opt/focuscall/registry.json

Funktion provision_container(user_id, agent_id, llm_key, bot_token, llm_provider):
1. Registry laden (JSON), naechsten Port zuweisen (base 42000, auto-increment next_port)
2. Workspace erstellen: WORKSPACE_BASE/{user_id}/{agent_id}/
3. config.toml.tmpl lesen, rendern mit string.Template: $USER_ID, $AGENT_ID, $PORT, $LLM_PROVIDER (KEINE Keys im Template!)
4. Gerendertes config.toml nach Workspace schreiben
5. docker.from_env().containers.run():
   - name: fc-{user_id}-{agent_id}
   - image: zeroclaw:latest
   - environment: { ZEROCLAW_API_KEY: llm_key, TELEGRAM_BOT_TOKEN: bot_token, ZEROCLAW_CONFIG_DIR: /workspace }
   - volumes: { workspace_path: { bind: /workspace, mode: rw } }
   - security_opt: ["no-new-privileges"]
   - cap_drop: ["ALL"]
   - read_only: True (root filesystem)
   - mem_limit: "128m"
   - nano_cpus: 500000000 (= 0.5 CPUs)
   - restart_policy: { "Name": "unless-stopped" }
   - detach: True
   - tmpfs: { "/tmp": "size=10m" } (da read_only root, braucht ZeroClaw /tmp)
6. Registry aktualisieren: instances[f"{user_id}-{agent_id}"] = { port, status: "starting", created_at }
7. Health-Check Loop: 10 Versuche, alle 2s, GET http://container_ip:{port}/health
8. Bei Erfolg: Registry status auf "running" setzen
9. Bei Fehler: Container stoppen und entfernen, Registry status auf "error"
10. Return: { status, port, container_id }

Funktion deprovision_container(user_id, agent_id):
1. Container fc-{user_id}-{agent_id} stoppen (timeout=10) und entfernen
2. Workspace-Verzeichnis loeschen (shutil.rmtree)
3. Registry-Eintrag entfernen
4. Return: { status: "removed" }

Funktion list_containers():
- Registry lesen, alle Instanzen mit Status zurueckgeben

Thread-Safety: filelock oder fcntl fuer registry.json Zugriffe (concurrent provisioning).

**Dockerfile** (Multi-stage ARM64 ZeroClaw Build):
- Stage 1 "builder": FROM rust:1.77-alpine AS builder
  - apk add musl-dev git openssl-dev pkgconfig
  - git clone https://github.com/b0xtch/zeroclaw.git /build
  - WORKDIR /build
  - cargo build --release --target aarch64-unknown-linux-musl (fuer ARM64 static binary)
  - Falls cross-compile noetig: Alternativ FROM rust:1.77-bookworm mit cargo build --release (dynamisch gelinkt)
- Stage 2 "runtime": FROM debian:bookworm-slim
  - RUN apt-get update && apt-get install -y --no-install-recommends ca-certificates && rm -rf /var/lib/apt/lists/*
  - RUN groupadd -r zeroclaw && useradd -r -g zeroclaw zeroclaw
  - COPY --from=builder /build/target/release/zeroclaw /usr/local/bin/zeroclaw (Pfad anpassen falls aarch64 target)
  - RUN chmod +x /usr/local/bin/zeroclaw
  - ENV ZEROCLAW_CONFIG_DIR=/workspace
  - VOLUME /workspace
  - USER zeroclaw
  - HEALTHCHECK --interval=30s --timeout=5s CMD curl -f http://localhost:${PORT:-42000}/health || exit 1
  - ENTRYPOINT ["zeroclaw", "daemon"]
- Kommentar im Dockerfile: "Build on ARM64 host (Hetzner CAX21) or use buildx for cross-compile"
